# Visualisation

For the educational and interactive requirement given by the client, a method of giving visual feedback, which represents the sound heard, should be given to the user. This visualization had to be both real-time and easy to read.

## Design Description

The PyQT library has been chosen to be used for the real-time plotting. This has been chosen due to its ability to show real-time plots and the ease of creating a GUI with it, which makes testing more efficient. Other libraries considered where libraries like matplotlib and plotly. However, matplotlib is not suitable for real time updates and plotly added too much complexity for this project. 

The signal processing group of the project handles the waveform generation, which is given in the form of chunks of 4096 samples with a sample frequency of 48 kHz. This results in a resolution of:

$$
\Delta f = \frac{fs}{N} = \frac{48000}{4096} = 11.7 Hz
$$

This resolution results in a latency of:

$$
T = \frac{1}{\Delta f} = 85,3 ms
$$

This is within the requirement mention in [@ref naar POR]

Four different ways of visualizing the sound are being used.

The first is the waveform, represented in a time-versus-amplitude graph. This is used to show the waveform, which can be compared to the sound heard by the students.

The second visualization is the frequency domain representation of the wave. The frequency-domain representation is obtained using the Fast Fourier Transform (FFT) algorithm implemented in the SciPy library. The FFT efficiently computes the discrete-time fourier transform, defined as:

$$
y[k] = \sum_{n=0}^{N-1} e^{-2\pi j \frac{kn}{N}} x[n]
$$

However, the fftfunction alone will come with some spectral leakage, to minimize this a Hann window was applied prior to the fft computation. The Hann window was used because it is one of the most commonly used windows for general spectral analysis because of its good side-lobe suppression and frequency resolution. [@10.1109/PROC.1978.10837]

To improve the bin spacing of the FFT, which is currently 11.7 Hz, quadratic interpolation can be used. This is a method of fitting a parabola trought the peak and it neighbour bins. This results in a more accurate frequency estimation than the 11.7 Hz. [@10.3390/app6100306]

For this visualisation, a logarithmic frequency axis has been chosen. This has several reasons; first of all, musical notes are logarithmically divided so a lotharimic representation makes more sense. Secondly, a large range of the frequency axis is needed to show all the harmonics, especially for blockwaves and sawtooth waves. 

For these two visualisation, automatic scaling of the vertical axes is used to ensure that signals of varying amplitudes remain visible without requiring manual adjustment:

```python
    if np.max(np.abs(chunk)) > p1_ymax:                             
        p1_ymax = p1_ymax * 1.5
        p1.setYRange(-p1_ymax, p1_ymax)
    if np.max(fft_values) > p2_ymax:
        p2_ymax = p2_ymax * 1.5
        p2.setYRange(0, p2_ymax)
    if np.max(fft_values) > p3_fft_ymax:
        p3_fft_ymax = p3_fft_ymax * 1.5
        p3_fft.setYRange(0, p3_fft_ymax)
```

The third visualization used is the waterfall spectrogram. This spectrogram is created using the `scipy.signal` spectrogram function. However, this function is only able to create a waterfall plot of a complete signal. Therefore, a spectrogram is calculated for each incoming audio chunk. The resulting spectral data are appended to a rolling buffer, producing a continuously updating waterfall display. This visualisation in combined with the fft in the same window. 

Another visualization created is the note visualization. This visualization shows the notes that are being played by the theremin. For this, the frequency is transformed into MIDI notation via the following formula:

$$
n = 69 + 12 \log_2\!\left(\frac{f}{440}\right)
$$
the frequency that is used in this formula is determined by locating the frequency bin with the largest magnitude in the FFT spectrum.

The result of this formula is a number. To convert it to a note, we take the modulus of 12 of this number and use it as an index in an array of note labels. For the octave we divide the number by 12, subtract 1 and round it down to an integer:

```python
note_labels = ['C', 'C#', 'D', 'D#', 'E', 'F', 'F#', 'G', 'G#', 'A', 'A#', 'B']
note = note_labels[n % 12]
octave = n // 12 -1
```

To visualize these notes in a way to into a way that students without a musical background feel more familiar with has been done by representing the notes by viaulizing a piano and showing which key closed to the sound heard. To make this piano, simple white rectangles are drawn based on the minimal and the maximal midi value. The values chosen are:

```python
midi_min = 21 
midi_max = 108
```

These values are chosen according to the minimal and maximal midi values of a full piano. 


## Evaluation and Testing Method
For validation of the visualization software, several prerecorded WAV files were used as test signals. At the time of development, the signal-generation subsystem and UART communication interface were still under active development by another project subgroup. Therefore, audio files containing known waveforms were used as reproducible test inputs. 

Multiple waveforms have been tested and the resulting visualisation had been compared to the theoratical outputs. Also, a wavefile with alternating frequencies and amplitudes has been tested to make sure alternating frequencies would not be an issue for the code.
